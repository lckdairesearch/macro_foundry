Core Onboarding Graph

The normal path is read-heavy first, then gated, then monitored in test. Ambiguity blocks catalog creation. Dangerous published-series fixes take a separate branch.

Normal path

Researcher Research + DB gap check

Probe source, inspect existing catalog state, infer likely mapping.

->
Researcher Draft proposal

Summarize existing rows, new rows, assumptions, risks, and suggested derived series.

->
Reviewer Fact-check + schema fit

Read-only review of naming, source facts, and table-fit. May update workflow flags.

->
Human Gate 1 Onboarding approval

Required before any canonical catalog or feed mutation.

->
Executor Apply in test

Create or update approved catalog rows, source mappings, and feed config in test.

->
Executor Initial full-history ingest

Monitored broad backfill in test; tolerate late provider coverage, fail on trust issues.

->
Human Review test outcome

Accept as test-approved or send back for refinement.

->
Output Emit onboarding package

Durable handoff artifact for later promotion workflow.

Derived series are allowed as proposal suggestions, but they do not block new raw source onboarding.

Review loop

Reviewer Reject for refinement

Reviewer can request changes without touching catalog or observations.

->
Researcher Refine proposal

Update reasoning, facts, mapping, or ambiguity notes.

->
Reviewer Re-review

Maximum three cycles. After that, escalate unresolved issues to the human gate.

Ambiguity branch

Guardrail Identity ambiguity detected

Example: unclear default variant versus narrower sibling variant.

->
Policy Stay in proposal space

No draft canonical series row is created in the main catalog.

->
Human Resolve or approve assumption

Agent may suggest the likely interpretation, but cannot silently mint identity.

Dangerous correction branch

Researcher Published-series identity problem

Underspecified or wrong canonical variant discovered after publication boundary.

->
Reviewer Impact review

Check affected source mappings, feeds, observations, and derivations.

->
Human Gate 2 Dangerous correction approval

No auto-rename for published canonical series.

->
Executor Apply approved repair

Usually human-reviewed correction, often supersede-and-replace instead of silent rename.

Decision Ledger

These are the rules the graph now enforces.

Canonical identity

  • series.code is semantic and provider-agnostic.
  • Provider tickers stay in series_sources.external_code.
  • Default variants are allowed when omission is an intentional curation choice.
  • If methodological scope is materially ambiguous, the agent proposes instead of creating identity.

Agent boundaries

  • Researcher gathers facts, probes, and drafts/refines proposals.
  • Reviewer is read-only for catalog/data, but may update workflow state and flags.
  • Executor writes only after approval.
  • Reviewer refinement loop is capped at 3 tries.

Initial ingestion

  • First real run happens in test.
  • It is monitored, not fire-and-forget.
  • It should aim for full history or a broad backfill.
  • Later recurring runs switch to overlap-window incremental fetches.

Published-series correction

  • Published canonical series are effectively immutable to autonomous agents.
  • No auto-rename once a series crosses the publication boundary.
  • Dangerous fixes require impact review and human approval.
  • The default repair bias is toward supersede-and-replace over silent rename.

Operational Rules

Implementation notes that shape the future LangGraph build without overloading one prompt.

MCP database access

The graph should inspect real Postgres state during research and review.

  • Look up existing concepts, families, series, sources, feeds, and lineage.
  • Do not rely only on the model's internal memory.

Small context helpers

Keep prompts narrow to avoid context overflow.

  • Catalog lookup helper
  • Proposal drafting helper
  • Reviewer check helper
  • Correction impact helper
